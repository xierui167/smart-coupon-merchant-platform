# 压测与验证计划

## 目标

当前简历中涉及 QPS 和响应时间优化时，应尽量配合本地压测报告，而不是只停留在设计描述。压测目标不是证明项目达到生产级能力，而是让面试表达更可信。

## 建议压测接口

| 场景 | 接口 | 关注指标 |
| --- | --- | --- |
| 同步领券 | `POST /api/v1/coupon/receive/{templateId}` | QPS、平均响应时间、错误率、库存是否超发 |
| 异步领券 | `POST /api/v1/coupon/receive/v2/{templateId}` | QPS、平均响应时间、MQ 堆积情况、结果查询成功率 |
| 查询异步结果 | `GET /api/v1/coupon/receive/v2/result/{requestId}` | 平均响应时间、Redis 命中情况 |
| 批量发券 | `POST /api/v1/coupon/distribution-tasks` | 任务创建耗时、消费耗时、成功数、失败数 |

## 压测工具

可以选择其中一种：

- JMeter：适合生成图形化报告，面试展示直观。
- wrk：适合快速压 HTTP 接口。
- k6：脚本化程度高，适合保存压测脚本。

## 压测前准备

1. 准备固定库存的优惠券模板，例如库存 1000。
2. 准备足够多的测试用户 ID，避免单用户领取限制影响结果。
3. 清理 Redis 中相关模板缓存和领取记录。
4. 确认 RocketMQ、Redis、MySQL 都正常运行。
5. 记录机器配置，例如 CPU、内存、JDK 版本。

## 需要记录的数据

- 并发数
- 总请求数
- 成功数
- 失败数
- 平均响应时间
- P95 / P99 响应时间
- QPS
- 最终库存
- 用户券数量
- 是否出现重复领取
- 是否出现库存超发

## 推荐表达

如果压测数据还没有最终完成，简历和面试中建议这样表达：

> 当前本地项目已完成秒杀链路设计与基础验证，QPS 和响应时间数据作为压测目标持续补充中。

如果完成压测，可以这样表达：

> 在本地固定机器配置下，使用 JMeter 对异步领券接口进行压测，并发数为 X，总请求数为 X，接口平均响应时间为 X ms，QPS 为 X，最终库存与用户券记录一致，未出现超发。

## 注意事项

- 不要把本地压测数据说成线上生产数据。
- 不要只写 QPS，不写机器配置和压测条件。
- 如果压测没有覆盖完整链路，就说明是接口级压测，不要说成系统级压测。
