# 面试介绍与追问

## 1 分钟项目介绍

智慧券商家服务平台是我用于复习 Java 后端和准备秋招的核心项目，主要面向商家优惠券营销场景，包含优惠券模板管理、预约提醒、秒杀领券、批量发券、核销、业务日志和定时任务兜底等能力。

技术上使用 Spring Boot 3、MySQL、Redis、RocketMQ、XXL-Job、Sa-Token 和 bizlog-sdk。项目重点不是简单 CRUD，而是围绕高并发领券、异步解耦、缓存一致性和业务可追踪性做了设计和优化。

## 3 分钟技术亮点

第一，秒杀领券链路中，我用 Redis Hash 缓存模板库存、状态和有效期，通过 Lua 脚本原子完成库存预扣和用户领取次数记录，降低并发下库存超发和重复领取风险。

第二，领券和批量发券都使用 RocketMQ 做异步解耦。秒杀请求提交后先返回 requestId，后续由 MQ 消费者完成数据库扣减、用户券写入和结果回写。批量发券场景则通过 EasyExcel 流式解析用户列表，避免接口同步阻塞。

第三，项目引入 XXL-Job 做定时兜底，例如优惠券自动上架、过期失效和库存同步补偿。Redis 用于热点数据，MySQL 作为最终可信数据源，异常场景下通过定时任务保证最终一致性。

第四，工程化方面，我用责任链拆分业务校验，用自定义注解和 AOP 抽象防重复提交与 MQ 消费幂等，并引入 bizlog-sdk 记录核心操作日志，方便后续排查和审计。

## 高频追问

### 为什么秒杀不用直接扣 MySQL？

高并发下所有请求直接打 MySQL，会让数据库成为瓶颈。Redis 更适合承载热点库存，Lua 能保证校验和扣减的原子性。MySQL 仍然作为最终数据源，消费端落库时通过条件扣减兜底，避免最终库存为负。

### Redis 扣了库存，MQ 发送失败怎么办？

接口层需要在 MQ 发送失败时回滚 Redis 预扣库存和用户领取次数记录，并把请求结果标记为失败。这样可以避免 Redis 中库存被错误占用。

### MQ 消息重复消费怎么办？

通过自定义 `@NoMQDuplicateConsume` 注解和 Redis 状态标记控制消费幂等。消费中、已消费和消费失败分别对应不同状态，避免重复消息导致重复发券。

### 怎么防止用户重复提交？

接口层通过自定义 `@NoDuplicateSubmit` 注解结合 Redis 窗口和 Redisson 锁控制重复提交，业务层再通过用户领取次数和数据库唯一约束做兜底。

### Bitmap 提醒有什么优势？

提醒类型和提醒时间组合很多，如果每种提醒都单独存一行，会增加存储和查询成本。使用一个 Long 类型位图就能记录多种提醒状态，判断和取消都可以通过位运算完成。

### RocketMQ 和 XXL-Job 有什么区别？

RocketMQ 更适合事件驱动，比如用户提交领券请求、商家创建批量发券任务。XXL-Job 更适合周期性补偿，比如过期券扫描、库存同步、定时上架。两者不是替代关系，而是分别处理异步事件和周期任务。

### ShardingSphere 是否已经启用？

当前主链路没有默认启用 ShardingSphere，而是保留了分库分表设计方案。原因是项目当前阶段更关注核心链路稳定和面试演示，过早引入分库分表会增加调试和事务复杂度。后续如果用户券数据量上来，会优先对用户券表和模板表按模板 ID、用户 ID 做复合分片。

### 为什么 GitHub 不放完整源码？

完整代码基于付费课程项目做学习、复现和二次开发，出于版权原因不公开完整源码。公开仓库主要展示我的项目理解、核心设计、技术取舍和复盘材料。面试时可以现场讲解核心链路和关键实现思路。

## 真实边界

- 当前项目更适合作为 Java 后端实习主力项目，不应该包装成完整生产级 SaaS。
- QPS 和响应时间需要通过压测报告支撑，不能把设计目标说成真实线上数据。
- ShardingSphere 是方案预留，不是主链路默认启用。
- 公开仓库不包含完整源码，主要用于展示项目理解和技术复盘。
